## Story ŌĆö Original Text

### Description

Als **dashboardgebruiker**
wil ik de vastgelegde ITS-duiding kunnen raadplegen per gegevenstype,
zodat ik zie wat al afgedekt is en wat ontbreekt.

**Voor wie doen we dit**
- Voor dashboardgebruikers (bronhouders/wegbeheerders)

**Wat is wenselijk (functioneel)**
- Overzichten zijn herleidbaar naar concrete publicaties
- Alleen door Moderator vastgelegde duiding telt mee (geen aannames)
- Het dashboard kan per verordening het **totaal aantal wettelijke verplichtingen** bepalen
- Het dashboard kan per verplichting bepalen of deze is **afgedekt door ├®├®n of meer publicaties**
- De bestaande dashboardstructuur (ITS/MMTIS / RTTI / SRTI / SSTP + voortgangsbalken) kan ongewijzigd blijven

### Acceptance Criteria

Het register kan per verordening bepalen:
- het totaal aantal wettelijke verplichtingen
- het aantal verplichtingen dat is **afgedekt** (ja/nee)

Een wettelijke verplichting geldt als **afgedekt** wanneer:
- minimaal ├®├®n publicatie is gekoppeld aan het juiste gegevenstype
- de vastgelegde scope overeenkomt met de verplichting
- de koppeling met de publicatie is vastgelegd door een Moderator

Wettelijke verplichtingen zonder bijbehorende publicatie gelden als **niet afgedekt**.

De berekening van percentages sluit aan op de bestaande dashboardopzet en is volledig herleidbaar naar concrete publicaties.

## Analysis

### Backend API ŌĆö revised structure (branch: feature/108265-regulations-dashboard-endpoint)

The backend team restructured `/data-regulations` to return a fully nested response with coverage embedded directly.
The original plan (separate `/obligations` endpoint + ITS 5th card) is obsolete.

```
GET /data-regulations ŌåÆ DataRegulationDto[]
GET /data-regulations/{regulationType} ŌåÆ DataRegulationDto

DataRegulationDto:
  regulationType: RegulationEnum (MULTIMODAL | REALTIME | ROAD_SAFETY | TRUCK_PARKING)
  dataItems: DataItemDto[]
    id: UUID
    description: string          ŌåÉ free-text label (Dutch)
    sourceLocation: string       ŌåÉ e.g. "1.1.d.vii"
    sourceDescription: string    ŌåÉ section heading
    sourceCategory: string       ŌåÉ sub-category
    obligations: ObligationDto[]
      id: UUID
      startDateNewData?: date
      startDateExistingData?: date
      obligationType?: EU | NATIONAL
      networkType?: TEN_T_PLUS | ...
      modality?: ...
      publications: DataPublicationReferenceDto[]   ŌåÉ coverage is HERE
        id: UUID
        name: string
```

### Key architectural change vs original plan

| | Original plan (obsolete) | Current plan |
|---|---|---|
| ITS card | Add 5th card from `/obligations` | **No 5th card** ŌĆö overview shows 4 cards unchanged |
| Coverage source | `hasPublications: boolean` (flat) | `obligations[].publications.length > 0` (nested) |
| Separate endpoints | `/data-regulations` + `/obligations` | Only `/data-regulations` |
| ObligationRepository | Needed | **Deleted** ŌĆö no longer needed |
| Detail API call | getAll() then filter | `GET /data-regulations/{type}` (single type) |

### Coverage rule (confirmed)

A **dataItem** is covered if **any** of its obligations has Ōēź1 publication linked.

### Navigation

Clicking a publication on the detail page navigates to `/publicaties/:id` (existing `PublicationsDetailComponent`).

### Decisions made

- **No ITS card** ŌĆö the overview shows only the 4 regulation types returned by `/data-regulations`
- **GET /data-regulations/{type}** for the detail page (not getAll + filter)
- **dataItem-level coverage**: covered if any obligation has Ōēź1 publication
- **item.description** shown as raw text (not translated enum key)
- **ObligationRepository/Service deleted** ŌĆö completely replaced by new structure
