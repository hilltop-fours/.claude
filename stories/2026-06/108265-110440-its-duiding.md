### Description

Als **dashboardgebruiker**
wil ik de vastgelegde ITS-duiding kunnen raadplegen per gegevenstype,
zodat ik zie wat al afgedekt is en wat ontbreekt.

**Voor wie doen we dit**
- Voor dashboardgebruikers (bronhouders/wegbeheerders)

**Wat is wenselijk (functioneel)**
- Overzichten zijn herleidbaar naar concrete publicaties
- Alleen door Moderator vastgelegde duiding telt mee (geen aannames)
- Het dashboard kan per verordening het **totaal aantal wettelijke verplichtingen** bepalen
- Het dashboard kan per verplichting bepalen of deze is **afgedekt door ├®├®n of meer publicaties**
- De bestaande dashboardstructuur (ITS/MMTIS / RTTI / SRTI / SSTP + voortgangsbalken) kan ongewijzigd blijven

### Acceptance Criteria

Het register kan per verordening bepalen:
- het totaal aantal wettelijke verplichtingen
- het aantal verplichtingen dat is **afgedekt** (ja/nee)

Een wettelijke verplichting geldt als **afgedekt** wanneer:
- minimaal ├®├®n publicatie is gekoppeld aan het juiste gegevenstype
- de vastgelegde scope overeenkomt met de verplichting
- de koppeling met de publicatie is vastgelegd door een Moderator

Wettelijke verplichtingen zonder bijbehorende publicatie gelden als **niet afgedekt**.

De berekening van percentages sluit aan op de bestaande dashboardopzet en is volledig herleidbaar naar concrete publicaties.

## Analysis

### Two separate backend endpoints

**`GET /data-regulations`** ŌĆö powers the existing 4 cards:
- Returns 4 grouped regulations (MULTIMODAL, REALTIME, ROAD_SAFETY, TRUCK_PARKING) + empty stubs for others
- Has `hasPublications: boolean` per data item ŌĆö real coverage data
- Marked deprecated in code but still the only source of coverage info
- **No ITS** ŌĆö completely absent from this endpoint

**`GET /obligations`** ŌĆö 131 flat obligations:
- Includes ITS obligations (`regulationType: 'ITS'`) with free-text Dutch descriptions
- Also includes MULTIMODAL and REALTIME obligations (overlap with data-regulations but different structure)
- **No coverage data** ŌĆö no `hasPublications` field anywhere
- The `Obligation` domain entity has a `publications` set (many-to-many), but `ObligationMapper` does NOT expose it in the API response yet

### Backend commit context (story #108264, merged 2026-02-17)

The previous story added `obligationIds` to `DataPublicationDto` so Moderators can link publications to obligations. This is the foundation. Story #108265 (this one) is the dashboard that shows the result. However, the `/obligations` endpoint still doesn't return coverage ŌĆö that's a future BE story.

### What the FE can build now

- **Keep `/data-regulations`** for the existing 4 cards ŌĆö don't touch it
- **Add a new separate data layer** (`ObligationService` + `ObligationRepository`) calling `/obligations`
- Filter ITS obligations, show as a 5th card with `0 / 131` coverage (all uncovered until BE adds coverage)
- Detail page shows all 131 ITS obligation descriptions as free text (not translation keys)

### Decisions made

- **Separate repositories** ŌĆö do not merge `/data-regulations` and `/obligations` into one repo
- **All 131 rows** on detail page (not deduplicated by dataItem.id)
- **0% coverage** for ITS is acceptable ŌĆö it's honest about the current state
- **Future BE story needed** to add `hasPublications` or `covered` to `ObligationDto`
